CraftMeshExporter - 3MF export
==============================

Adds .3mf (3D Manufacturing Format) as a third export format alongside STL and OBJ.
Any combination of the three can be selected; all selected formats are written in one
export run.

New file
--------
ThreeMfWriter.cs
    ThreeMfWriter    - builds the 3MF package: [Content_Types].xml, _rels/.rels,
                       3D/3dmodel.model. Deduplicates vertices into an indexed vertex
                       list, drops triangles that degenerate after welding, writes
                       unit="millimeter" and craft metadata.
    MinimalZipWriter - a small ZIP writer (stored/uncompressed entries, CRC-32, central
                       directory, EOCD). This exists so the mod does NOT depend on
                       System.IO.Compression.dll, which KSP's Mono runtime does not
                       reliably expose to plugins. Same reasoning as the COM/dynamic
                       avoidance in ShortcutResolver.

Output
------
    SaturnV_printable.3mf
    SaturnV_printable_3MF_instructions.txt

(In Flight, the _flight_current_printable suffix is used, as with the other formats.)

Notes
-----
The 3MF is geometry only, like the STL and OBJ; no textures, no per-part color.
The ZIP byte layout (local headers, central directory, EOCD, CRC-32) was verified
against a standard ZIP reader before shipping.


===============================================================================
STEP (.stp) export
===============================================================================

Adds .stp (ISO 10303-21, schema AUTOMOTIVE_DESIGN / AP214) as a fourth export
format. Any combination of STL / OBJ / 3MF / STEP can be selected; all selected
formats are written in one export run.

READ THIS FIRST
---------------
STEP is a boundary-representation CAD format. A KSP craft is a triangle mesh.
The only faithful way across that gap is a FACETED B-rep: every triangle becomes
its own flat ADVANCED_FACE, on its own PLANE, bounded by three EDGE_CURVEs.

That is valid STEP and it opens in Fusion 360, SolidWorks, FreeCAD, Onshape.
It is NOT a "real" CAD model. No NURBS surfaces, no analytic cylinders. CAD will
show thousands of tiny planar facets, because that is all a mesh contains, and no
converter can invent the smooth surfaces back.

It is also large: roughly 17 STEP entities per triangle. A 100k-triangle craft is
on the order of 1.7M lines and 100+ MB, and slow to import. Decimate in Blender or
MeshLab first if that is unmanageable. The exporter emits a status warning above
50,000 triangles.

Most slicers cannot read STEP. Use STL or 3MF for printing; use STEP for CAD work.

New file
--------
StepWriter.cs
    Streams the file directly to a StreamWriter rather than building it in a
    StringBuilder. A large craft would otherwise assemble a several-hundred-MB
    string in memory and take KSP down with it.

    Shares CARTESIAN_POINTs, VERTEX_POINTs, and EDGE_CURVEs between the triangles
    that use them. This is not cosmetic: unshared output is roughly double the
    entities, and CAD kernels are much happier when two faces meeting at an edge
    reference the same edge rather than two coincident ones. Edges are stored in
    canonical (low vertex id first) order; a triangle traversing one backwards gets
    an ORIENTED_EDGE with .F., which is precisely what that flag is for.

    Emits OPEN_SHELL + SHELL_BASED_SURFACE_MODEL, NOT CLOSED_SHELL +
    MANIFOLD_SOLID_BREP. Deliberate: a KSP craft is many overlapping part meshes and
    is not watertight (the README says so). A STEP file that declares itself a closed
    solid while not being one is rejected outright by CAD kernels. A surface body is
    the honest description of what this actually is.

    Drops triangles that degenerate after vertex welding, since a zero-area face has
    no valid plane normal and cannot be represented.

Verification
------------
The emitted STEP was checked for reference integrity (no dangling entity refs, no
duplicate ids), valid REAL formatting (STEP requires "10.0", not "10"), and correct
topology sharing: a tetrahedron comes out with 4 vertices, 6 edges, 4 faces, giving
Euler characteristic V-E+F = 2 as it should.


===============================================================================
3MF: one object per part, with a color palette
===============================================================================

The 3MF export can now emit one <object> per KSP part inside a <components>
assembly, with a <basematerials> palette giving each part its own color, instead
of a single merged mesh. On by default; the sub-toggle under "Export 3MF" turns it
off to get the old merged behavior back.

A slicer opening this shows each part as its own selectable body: deletable,
movable, assignable to its own extruder. STL structurally cannot express that.

About the colors, stated plainly
--------------------------------
They are SYNTHETIC and decorative. A KSP part is textured, not flat-colored, so
there is no single honest "color of this part" to read out of the game. The palette
exists only so adjacent parts are visually distinguishable. It is NOT the part's
real in-game appearance and should not be read as a paint plan.

Hues advance by the golden angle rather than being evenly divided, so consecutively
numbered parts (usually physically adjacent on the craft) land far apart on the
color wheel, and the palette stays well separated at any part count. Saturation and
value alternate so neighbors stay distinct even once hues start to crowd on very
large craft. Deterministic in the part ordinal, so re-exporting the same craft gives
the same colors instead of reshuffling them.

How part attribution survives the pipeline
------------------------------------------
Triangle gained an int PartIndex field, stamped by MeshCollector.

This is on the struct rather than in a parallel array on purpose. MeshCleanup's
RemoveBadTriangles and RemoveDuplicateTriangles delete list elements, and a side
array indexed in parallel would silently desync from the triangles it describes.
Carried on the struct, attribution survives cleanup for free and MeshCleanup did not
need to change at all.

Consequence worth knowing: RemoveDuplicateTriangles collapses coincident triangles
from overlapping parts down to whichever it sees first, so a shared face is
attributed to one part rather than duplicated. That is the desired behavior.

Verification
------------
The emitted per-part model was checked against the 3MF core spec rules: resource ids
unique across objects AND basematerials (they share one id space), every pid resolves
to a real basematerials and every pindex is in range, every object carries a mesh XOR
components but never both, component objectids resolve with no self-reference, the
build item resolves, and each object's triangle indices stay inside that object's own
independent vertex space.
